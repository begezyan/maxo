# Отчет о различиях между `maxo` и `aiogram` / `aiogram_dialog`

## Введение

Фреймворк `maxo` является асинхронным решением для разработки ботов на платформе `max.ru` и, согласно `README.md`, основан на `aiogram`, `aiogram_dialog` и `aiogram-webhook`. Данный отчет анализирует ключевые различия между `maxo` и его базовыми компонентами (`aiogram` и `aiogram_dialog`), фокусируясь на импортах, инициализации, обработке событий и структуре диалогов.

## 1. Общее сравнение (по примерам)

### Инициализация и Запуск

-   **`maxo`**: Использует свой класс `LongPolling` для запуска, который вызывается синхронно. `Dispatcher` не содержит логики получения обновлений.
-   **`aiogram`**: Использует асинхронный метод `dp.start_polling()` внутри `Dispatcher`, который сам управляет циклом получения обновлений.

### Обработчики и Ответы

-   **`maxo`**: Вводит концепцию **Фасадов** (`MessageCreatedFacade`). Обработчики получают объект фасада, который абстрагирует и упрощает отправку ответов (`facade.answer_text(...)`).
-   **`aiogram`**: Использует методы, доступные непосредственно на объектах событий (`message.answer()`, `callback_query.answer()`).

## 2. Глубокое сравнение компонентов

Этот раздел углубляется в архитектурные различия ключевых компонентов фреймворков.

### Типы и Сериализация
-   **Пути**: `src/maxo/types/` vs `aiogram/aiogram/types/`
-   **Ключевые различия**:
    -   **Основа**: Типы в `maxo` — это, как правило, простые классы, похожие на dataclass. Типы в `aiogram` основаны на `pydantic.BaseModel`.
    -   **Валидация**: `aiogram` использует встроенные возможности `pydantic` для валидации, приведения типов и сериализации на уровне самих моделей. `maxo` делегирует всю эту логику централизованному объекту `Retort` из библиотеки `adaptix`.
    -   **Структура**: Типы `maxo` адаптированы под API `max.ru` и могут иметь другую структуру и названия полей в сравнении с Telegram Bot API.

### Enums (Перечисления)
-   **Пути**: `src/maxo/enums/` vs `aiogram/aiogram/enums/`
-   **Ключевые различия**:
    -   **Специфичность**: `maxo` содержит только перечисления, необходимые для `max.ru` API. Например, `maxo.enums.UpdateType` содержит значения `MESSAGE_CREATED`, `BOT_STARTED`, в то время как `aiogram.enums.UpdateType` содержит `MESSAGE`, `CALLBACK_QUERY` и т.д.
    -   **Полнота**: `aiogram` предоставляет исчерпывающий список перечислений, покрывающий все аспекты Telegram Bot API. `maxo` — это адаптированное подмножество.

### Errors (Ошибки)
-   **Пути**: `src/maxo/errors/` vs `aiogram/exceptions.py`
-   **Ключевые различия**:
    -   `maxo` определяет небольшую иерархию собственных ошибок (`MaxoError`, `MaxBotApiError`), специфичных для своего API клиента.
    -   `aiogram` имеет значительно более богатую иерархию исключений для обработки ошибок сети, API, фреймворка (например, `TelegramAPIError`, `ChatNotFound`, `FilterExpired`).

### Dispatcher и Router (Обработка и маршрутизация событий)

#### Архитектура

-   **`maxo`**:
    -   **Разделение ответственности**: `Dispatcher` отвечает только за обработку событий. Получение обновлений (поллинг, вебхуки) вынесено в отдельные классы (`LongPolling`, `WebhookEngine`) в модуле `transport`.
    -   **Маршрутизация по типам**: Маршрутизация основана на **типах** объектов обновлений. `Router` имеет словарь, сопоставляющий класс обновления (например, `MessageCreated`) с соответствующим наблюдателем (`Observer`).
    -   **Безопасность**: `Router` имеет встроенную машину состояний, которая запрещает изменение графа маршрутизации после запуска.

-   **`aiogram`**:
    -   **Монолитный `Dispatcher`**: `Dispatcher` является центральным компонентом, который управляет и получением обновлений, и их обработкой.
    -   **Маршрутизация по именам**: Маршрутизация основана на **строковых именах** событий (`"message"`, `"callback_query"`).

#### Механизм "Фасадов" в `maxo`

Это ключевое архитектурное отличие `maxo`, реализованное через `FacadeMiddleware`. Мидлварь извлекает объект-фасад из свойства `.facade` каждого объекта обновления и внедряет его в контекст события, делая доступным в обработчиках.

### Routing Sub-components (Компоненты маршрутизации)
-   **`updates`**: `src/maxo/routing/updates/` vs `aiogram/aiogram/types/update.py`
    -   **`maxo`**: Определяет десятки специфичных классов для каждого события. Каждый класс содержит свойство `.facade`.
    -   **`aiogram`**: Имеет один универсальный класс `Update`, содержащий опциональные поля для каждого типа события.
-   **`signals`**: `src/maxo/routing/signals/`
    -   Это **уникальная для `maxo` концепция**. Сигналы (`BeforeStartup`, `AfterShutdown`) используются для событий жизненного цикла и обрабатываются тем же механизмом маршрутизации, что и обычные обновления. В `aiogram` для этого используются отдельные наблюдатели `startup` и `shutdown`.

### FSM (Finite State Machine)

-   **`FSMContext`**, **`State`**, **`StatesGroup`**: Реализации в `maxo` являются практически прямыми копиями кода `aiogram`.
-   **Ключевое различие** в `FSMContextMiddleware`:
    -   **`maxo`**: Использует **фиксированную стратегию** хранения состояния. Ключ всегда формируется из `(bot_id, chat_id, user_id)`.
    -   **`aiogram`**: Предоставляет **гибкие стратегии** (`USER_IN_CHAT`, `CHAT` и др.) через параметр `fsm_strategy` в `Dispatcher`.

### Filters (Фильтры)
-   **`maxo`**: Упрощает стандартные фильтры. Например, из `Command` фильтра убрана поддержка `MagicFilter` и логика deeplink, которая вынесена в отдельный фильтр `Deeplink`.
-   **`aiogram`**: Предлагает более мощные "все-в-одном" фильтры, сочетающие несколько функций.

### Middlewares (Мидлвари)
-   **`maxo`**: Использует **"луковичный" (onion) паттерн**, где мидлварь получает функцию `next` для вызова следующего слоя.
-   **`aiogram`**: Использует **паттерн "обертки"**, где мидлварь получает `handler` (следующий слой) и оборачивает его вызов.

### Bot (Объект бота) и Transport
-   **Пути**: `src/maxo/bot/` и `src/maxo/transport/` vs `aiogram/client/` и `aiogram/dispatcher/dispatcher.py`
-   **Ключевые различия**:
    -   **HTTP-клиент**: `maxo` использует свой `MaxApiClient` на базе `unihttp`. `aiogram` использует заменяемую систему сессий (`BaseSession`), по умолчанию на `aiohttp`.
    -   **Сериализация**: `maxo` построен на `adaptix`, `aiogram` — на `pydantic`.
    -   **API Методы**: `maxo` использует метапрограммирование (`bind_method`) для компактного определения методов. `aiogram` явно определяет каждый метод в классе `Bot`.
    -   **Транспорт**: `maxo` четко отделяет транспортный уровень (`LongPolling`) от `Dispatcher`. В `aiogram` логика поллинга встроена в `Dispatcher`.

### Integrations и Utils
-   **`maxo`**: Содержит директорию `integrations` с готовыми решениями для `dishka` (Dependency Injection) и `magic_filter`. В `utils` ключевым элементом является реализация паттерна "Фасад", аналога которой нет в `aiogram`. `KeyboardBuilder` в `maxo` является аналогом `aiogram.utils.keyboard.InlineKeyboardBuilder`, но с немного другим API.
-   **`aiogram`**: Не предоставляет встроенных интеграций в ядре, но является достаточно гибким для их самостоятельного подключения.

---

## 3. Итоговое заключение

`maxo` не является простым ребрендингом `aiogram`. Это **основательный форк с собственным видением**, который заимствует проверенные концепции `aiogram` (FSM, структура диалогов), но кардинально перерабатывает ядро (Dispatcher, Middlewares, Bot Object) в сторону более строгих, современных и узкоспециализированных паттернов.

-   **`aiogram`** остается универсальным, чрезвычайно гибким и мощным фреймворком, предоставляющим разработчику полный контроль.
-   **`maxo`** делает ставку на **упрощение и строгость**: вводит фасады для унификации ответов, использует более современный паттерн мидлварей, обеспечивает безопасность графа роутеров и фиксирует некоторые конфигурации (например, FSM стратегию), чтобы снизить порог вхождения и уменьшить количество возможных ошибок для разработчиков, работающих с платформой `max.ru`.
